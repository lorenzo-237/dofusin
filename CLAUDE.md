# CLAUDE.md

Guidance for Claude Code (and other agents) working in this repo.

## What this is

DofusIn (repo/package name stayed `dofus-dispo`, product name is `DofusIn`): a desktop app (Tauri) that lets players mark one of their game characters as "available to help" for the day, and lets other players search for available helpers. Full spec in `project/cahier_charges.md` (French). The UI design comes from a Claude Design handoff extracted in `project/handoff_extracted/application-dofus-dispo/project/` — `AppShell Mobile.dc.html` is the source of truth for layout/copy/behavior (the only view that was retained; ignore the "Candy"/"Warm" desktop variants in that folder, they were abandoned explorations).

## Stack

- React 19 + TypeScript, Vite, Tailwind CSS v4
- shadcn (style `base-rhea`, built on `@base-ui/react`, not Radix) — components live in `src/components/ui/`
- TanStack Router, file-based (`src/routes/`), routes compiled to `src/routeTree.gen.ts`
- Tauri 2 for the desktop shell (`src-tauri/`), window sized to the mobile design (~430×900), native title bar disabled (`decorations: false`) in favor of a custom-styled one

## Commands

- `npm run dev` — Vite dev server (also generates/updates `routeTree.gen.ts`)
- `npm run build` — `tsc -b && vite build`
- `npm run typecheck` — `tsc --noEmit`
- `npm run lint` / `npm run format`
- `npm run tauri dev` / `npm run tauri build` (if you need the actual desktop shell, not just the web view)

`src/routeTree.gen.ts` is generated by the `@tanstack/router-plugin` Vite plugin (see `vite.config.ts`) and is gitignored. If it's missing (fresh clone) or `router.tsx` fails to import it, run `npm run dev` or `npx vite build` once to regenerate it before typechecking.

## Verification

Verify changes with `npm run typecheck`, `npm run lint`, and `npm run build` only. **Do not use chromium-cli, Playwright, or any other browser-automation tool against this app** — the user tests it themselves manually; don't launch a browser session on their behalf.

## Architecture

### Routing

File-based TanStack Router under `src/routes/`:
- `__root.tsx` — mounts `AuthProvider`, `MobileShell`, the sonner `Toaster`.
- `login.tsx` — redirects to `/` if a session already exists.
- `onboarding.tsx` — redirects to `/login` if there's *no* session (inverse of `login.tsx`); shown once, right after a brand-new Discord account's first login, to collect the first character the cahier des charges requires.
- `_authenticated.tsx` — pathless layout route; `beforeLoad` redirects to `/login` if there's no session. Renders the scrollable `<Outlet/>` and the bottom nav; the page title/header lives in the always-on `TitleBar` (see [Custom title bar](#custom-title-bar)), not in this route. Also redirects to `/onboarding` once `charactersLoaded && characters.length === 0` — this can't live in `beforeLoad` like the session check (characters live in React state, not the plain `auth-store`), so it's a `useEffect` in the route component instead.
- `_authenticated/index.tsx` (Accueil), `characters.tsx`, `jobs.tsx`, `availability.tsx`, `search.tsx`.

Auth guards read a plain module-level store (`src/lib/auth-store.ts`), not React state — `beforeLoad` runs outside components, so it can't use `useAuth()`. `AuthProvider` (`src/context/auth-context.tsx`) subscribes to that same store via `useSyncExternalStore` to expose it reactively (`useAuth()`), and also owns `characters`/`jobs`/`availabilities`/`jobAvailabilities` plus all their CRUD actions so screens stay thin.

**Auth is Discord OAuth2 only — no password is ever stored.** `LoginForm` is a single "Se connecter avec Discord" button calling `useAuth().loginWithDiscord()`; there's no separate register step (a fresh Discord account just lands in `/onboarding` instead, see above). The full flow — a local loopback HTTP server (`tiny_http`, `src-tauri/src/oauth.rs`) since Discord's OAuth2 rejects custom URI schemes as redirect_uri (`http://` only), `state` CSRF check, the `isTauri() && VITE_API_STRATEGY === "http"` gate that skips the whole browser round-trip in favor of a mock session everywhere else (plain browser tab, or mock strategy even inside Tauri) — is documented in `docs/discord-auth.md`. `AuthProvider` owns the `oauth://callback` event listener (registered once) and the pending-login promise (a ref resolved/rejected by that listener) that `loginWithDiscord()`/`cancelDiscordLogin()` coordinate through.

JWTs expire after 30 days (`dofusin-api/src/lib/jwt.ts`) — `src/lib/session-expiry.ts` is a tiny module-level pub/sub (same pattern as `auth-store.ts`) that `http-api-client.ts`'s plain `request()` function calls into (`notifySessionExpired()`) whenever an authenticated call (one that carried a token) comes back `401`, since a plain function can't reach the router/React state directly. `AuthProvider` is the sole subscriber (`onSessionExpired`): it logs out, shows a toast, and redirects to `/login`, so an expired session actually kicks the user out instead of leaving the app visually "logged in" while every request silently fails. Guarded by `getSession()` at the call site so the burst of parallel requests fired on mount only triggers this once, not once per failed request.

`src/lib/last-selection-store.ts` (plain `localStorage` get/set, no subscription needed — only read once per component mount) remembers the last server picked (shared across Mes personnages, Disponibilité, Recherche, inscription) and the last job picked (shared across Mes métiers and Recherche > Métiers), so those screens default to the user's last choice instead of always resetting to `SERVERS[0]`/`JOBS[0]`. Every `ServerSelect`/`JobSelect` consumer initializes its state from `getLastServer()`/`getLastJob()` and calls `setLastServer()`/`setLastJob()` alongside its own `onValueChange`/`onChange` handler — there's no automatic global sync, each screen opts in explicitly.

The Disponibilité screen (`_authenticated/availability.tsx`) forces a server selection first (like Recherche) and scopes both tabs to it: "Personnages" filters `characters` by `character.server`, "Métiers" filters `jobs` by `job.server` — each server's characters/jobs get independent availability + price. Character/job *configuration* (as opposed to availability) used to be tabs on one `/characters` page but is now two separate routes/burger-menu entries — `_authenticated/characters.tsx` ("Mes persos", `CharacterList`/`CharacterForm`) and `_authenticated/jobs.tsx` ("Mes métiers", `JobList`/`JobForm`) — split for clarity once both grew enough content to feel cluttered together. Each still owns its own server picker (not shared state between the two routes, same as `Server`/`JobSelect`'s general last-selection pattern below) and `CharacterForm`/`JobForm` take that `server` as a prop instead of asking for it again; `JobForm`'s character picker only lists characters already scoped to that server. `components/availability/empty-state-reminder.tsx`'s two call sites in `availability.tsx` now point `to` different routes (`/characters` vs `/jobs`) since there's no single combined destination anymore.

### Data layer: strategy pattern (mock vs real API)

Everything that touches data goes through the `ApiClient` interface (`src/lib/api/api-client.ts`). Two implementations:
- `MockApiClient` (`mock-api-client.ts`) — default. Fully client-side, state in `localStorage`, seeded with fictive helpers (`mock-data.ts`) so search results look populated even for a fresh account.
- `HttpApiClient` (`http-api-client.ts`) — real `fetch` calls against the Express API described in `project/cahier_charges.md`.

`getApiClient()` (`src/lib/api/index.ts`) picks the implementation based on `VITE_API_STRATEGY` (`mock` | `http`, see `.env.example`). **Never import `MockApiClient`/`HttpApiClient` directly outside that factory** — components and the auth context only depend on the `ApiClient` interface, so switching strategies is a `.env` change, not a code change.

Several things extend beyond the literal cahier des charges, all intentional and user-approved:
- `Character` carries a `level` field (the documented API only has `name`/`server`/`class`).
- `Job` (`getJobs`/`setJob`/`deleteJob`) is a standalone resource not in the cahier des charges: professions are per-account-per-server in the game (every character a user has on a given server shares the same job levels), not per-character, so it can't be modeled as a character field. `setJob` upserts by `(server, job)`. `Job.characterId` is *not* an ownership relationship — it's just which of the account's characters on that server to display/whisper alongside the job (picked explicitly in `JobForm` via `CharacterSelect`), since the level itself still applies account-wide.
- `ApiClient.getMyAvailabilities` / `HttpApiClient`'s `GET /availability/me` aren't in the cahier des charges — the frontend needs to know which of the current user's characters are already available today, and no documented route provides that. See the comment in `http-api-client.ts`.
- `JobAvailability` (`getMyJobAvailabilities`/`setJobAvailability`/`removeJobAvailability`) mirrors `Availability` but keyed by `jobId` instead of `characterId` — a job can be "available for hire" (with its own free/paid + price) independently of any character, for the same reason `Job` itself is a standalone resource.
- `searchJobHelpers`/`GET /job-helpers` is a second, separate search endpoint (not in the cahier des charges, which only documents `/helpers`) for browsing jobs marked available for craft — `JobSearchFilters` requires both `server` and `job` (no "browse everything" mode, unlike character search where server is required but class isn't).
- `getStaleAvailabilities`/`reactivateAvailabilities` (and the `...JobAvailabilities` mirrors) power the "reactivate everything as it was" banner on Accueil (`ReactivateBanner`): `Availability`/`JobAvailability` are one row per character/job (upserted, never deleted on expiry), so a "stale" row just means its `availableDate` isn't today — the previous free/price settings are still sitting right there. `AuthProvider` fetches both stale lists alongside the today-scoped ones, and filters an item out of the stale list wherever it's individually re-set (`setAvailability`/`setJobAvailability`) or deleted (`deleteCharacter`/`deleteJob`), so the banner's count can't drift from what's actually left to reactivate.
- **"Recherche intelligente" / `HelpRequest`**: a requester asks for a character (optionally filtered by class) or a specific job on a server; every currently-available matching account gets notified live over WebSocket. `src/lib/ws-client.ts` is a plain module-level singleton (same pattern as `auth-store.ts`) that connects only when `isTauri()`-independent — the gate is `VITE_API_STRATEGY === "http"`, since mock strategy has no backend to connect to — and reconnects on drop with a fixed delay. `AuthProvider` owns three lists (`incomingHelpRequests`, `myHelpRequests` as requester, `acceptedHelpRequests` as helper), REST-fetched once as a catch-up on login and then kept live by a single `onWsEvent` listener mapping `help-request:incoming/accepted/closed/resolved` to the matching list update. See `dofusin-api/docs/help-requests.md` for the full mechanics (xp/strikes, first-come-first-served atomicity, why the WS server pushes but never persists). The UI lives under the bottom nav's "Entraide" tab (`routes/_authenticated/help-requests.tsx`, `components/help-requests/`) — see `docs/notifications.md` for how the same `onWsEvent` listener also drives toasts and the branded popup window.
  `getMyHelpRequests`/`getAcceptedHelpRequests` (and their `dofusin-api` routes, `GET /help-requests/mine`/`/accepted`) are cursor-paginated (`HelpRequestPage = { items, nextCursor }`, not a bare array) — an unbounded history grows forever, and cursor pagination (the previous page's last item id) doesn't reshift like an offset would as new requests get created between calls. `AuthProvider` only tracks a cursor per list (`myHelpRequestsCursor`/`acceptedHelpRequestsCursor`, not exposed directly) plus `loadMoreMyHelpRequests`/`loadMoreAcceptedHelpRequests`, appending to the already-loaded array — `MyResponseList`'s "Charger plus" button (inside the collapsed "Historique" section) is the only UI trigger, since active (`OPEN`/`ACCEPTED`) requests are always within the first page in practice.

### Theming

`src/index.css` overrides the shadcn CSS variables with the Dofus-Dispo palette (cream background, forest green primary, gold accent, terracotta destructive) and swaps the font stack to Baloo 2 (`--font-heading`, via `@fontsource-variable/baloo-2`) and Quicksand (`--font-sans`, via `@fontsource-variable/quicksand`). Game data constants (`SERVERS`, `CLASSES`, `CLASS_COLORS`) live in `src/lib/game-data.ts`. Both `SERVERS` and `JOBS` follow the same shape: grouped by real category (`SERVERS_BY_CATEGORY`: Dofus/Dofus Retro/Dofus Touch; `JOBS_BY_CATEGORY`: Forgemagie/Récolte/Craft — 20 jobs total), sorted alphabetically within each category at the source (no client-side sort needed), with the flat `SERVERS`/`JOBS` tuples derived by spreading the categories in order (still the source of the `Server`/`JobName` unions) and `JOB_COLORS` assigned per category family. `ServerSelect`/`JobSelect` are two-step pickers: a `CategoryPicker` chip row (`src/components/shared/category-picker.tsx`, generic over the category union) narrows the list down to one category, then the `Select` below only lists that category's items — flattening 20 jobs or 20 servers into one dropdown was unusable. The category shown is *derived* from the current `value` prop via `serverCategory()`/`jobCategory()` (not separate internal state), so picking a category actually changes the underlying value too (jumps to that category's first item) — there's a single source of truth, no risk of the chip row and the dropdown disagreeing.

### Branding

`public/assets/icon.svg` (square app icon) and `public/assets/lockup.svg` (icon + "Dofus"+"In" wordmark, "In" in primary green) are the source brand assets. `src-tauri/icons/` is generated from `icon.svg` via `npx tauri icon public/assets/icon.svg -o src-tauri/icons` (regenerate the same way if the icon changes — don't hand-edit those PNGs/ICO/ICNS). The web favicon (`index.html`) points at `icon.svg` directly. `BrandHeader` (login/register) renders `lockup.svg` as-is. The custom title bar (`title-bar.tsx`) can't reuse the lockup because its green background would swallow the green "In" — it renders the icon plus a text wordmark with "In" at reduced opacity instead.

### Components

Organized by domain under `src/components/`: `auth/`, `characters/` (includes the "Mes métiers" job form/list, separate from the character form/list), `availability/`, `search/` (character search + a parallel `JobSearch*` family for the "Métiers" tab, each owning its own filters/fetch so it only runs while its `TabsContent` is mounted — Base UI unmounts inactive tab panels by default), `help-requests/` ("Entraide" — the create/incoming/my-response cards+lists for `HelpRequest`, see above), `home/`, plus `layout/` (shell, bottom nav, header) and `shared/` (things reused across domains: `ClassAvatar`/`ClassDot`, `ServerSelect`/`ClassSelect`/`JobSelect`/`CharacterSelect`, `CategoryPicker`, `CopyCommandButton`). `CopyCommandButton` (`characterName` + `price`) no longer copies straight to the clipboard — it opens a `Dialog` with an editable `Textarea` pre-filled with a whisper command ending in a `[Gratuit]`/`[Payant - Nk]` tag (derived from `price`), so the user can add a location/quest/note before copying; the message resets to the template every time the dialog re-opens (`onOpenChange`, not an effect). Route files under `src/routes/` stay thin — they wire `useAuth()`/`getApiClient()` to these components rather than containing UI logic themselves.

`CharacterForm` intentionally has no effect resyncing its fields to the `editingCharacter` prop — the parent (`routes/_authenticated/characters.tsx`) remounts it via `key={editingCharacter?.id ?? "new"}` instead (React's recommended way to reset state on identity change, and required by this repo's `react-hooks/set-state-in-effect` lint rule).

### Custom title bar

`layout/title-bar.tsx` replaces the native OS title bar (disabled via `decorations: false` in `tauri.conf.json`) so it can be styled with the app palette instead of system chrome — and, since there was no room to spare on a 430px-wide window, it also doubles as the in-app header: it reads the current page title the same way the old `ProtectedHeader` did (`useMatches()` + `staticData.title`, see `src/lib/router-static-data.d.ts`), centers it via absolute positioning (so it stays centered regardless of how wide the left icon / right button group are), and owns the burger menu (links to `/characters` "Mes persos", `/jobs` "Mes métiers", and `/settings` "Paramètres" — none has its own bottom-nav tab, see `BottomNav` below — plus theme switch + logout, only rendered when `useAuth().user` is set). Unlike the old split (Tauri-only title bar + separate always-rendered `ProtectedHeader`), this single bar is now always rendered — including in a plain browser tab during `npm run dev` — so the page title and burger menu stay reachable there; only the window controls are gated behind `isTauri()` (from `@tauri-apps/api/core`). Dragging works via the `data-tauri-drag-region` attribute (not JS); minimize/close call `getCurrentWindow()` from `@tauri-apps/api/window`, which requires the `core:window:allow-close`/`allow-minimize`/`allow-hide`/`allow-start-dragging` permissions explicitly granted in `src-tauri/capabilities/default.json` (`core:default` alone doesn't include them). No maximize button — the window is meant to stay phone-shaped. `_authenticated.tsx` no longer renders a header itself, just the scrollable `<Outlet/>` and `<BottomNav/>`.

`layout/bottom-nav.tsx` has 4 tabs — Accueil, Dispo, Recherche, Entraide (character/job config moved into the burger menu above, freeing up the slot) — and shows a small count badge on Entraide when `incomingHelpRequests.length > 0`.

### Settings and the system tray

`routes/_authenticated/settings.tsx` ("Paramètres", reached from the burger menu) currently holds a single preference — how the minimize button behaves — stored via `src/lib/settings-store.ts` (plain `localStorage`, read fresh at click-time by `title-bar.tsx` rather than kept in React state, since nothing needs to react live to it). `"taskbar"` (default) is the normal OS minimize; `"tray"` calls `getCurrentWindow().hide()` instead, so the window disappears entirely and is only reachable again through the system tray icon (`src-tauri/src/tray.rs`, `TrayIconBuilder` — requires the `tray-icon` feature on the `tauri` crate in `Cargo.toml`). The tray icon itself is always created on startup regardless of this preference (so "Afficher"/"Quitter" are always reachable) — the setting only changes what the minimize *button* does, not whether the tray icon exists. The tray menu and its click handler live entirely in Rust (`show_main_window`, called from both the "Afficher" menu item and a left-click on the icon); no new frontend-facing capability was needed for the tray itself, since the frontend never calls into it directly.

The main window's position and size persist across launches via `tauri-plugin-window-state` (`src-tauri/src/lib.rs`, plugin registered with no frontend-facing setup — it saves/restores automatically on window move/resize/close, no `invoke` calls needed). The `notification-popup` window is explicitly excluded (`.with_denylist(&["notification-popup"])`) since its position is always freshly computed bottom-right of the primary monitor at show time (see Notifications below) — letting the plugin "restore" a stale position for it would fight that logic.

Closing the app (the title bar's X, or the tray's "Quitter" — both routed through the same window-close flow, see `tray.rs`) is intercepted by `CloseConfirmDialog` (`components/layout/close-confirm-dialog.tsx`, mounted in `__root.tsx`) via `getCurrentWindow().onCloseRequested()`. Closing doesn't clear `Availability`/`JobAvailability` rows on its own — they'd just sit there dated today until the app is reopened — so this dialog warns about it and, on confirm, calls `useAuth().deactivateAll()` (the mirror-opposite of the existing `reactivateAll` — restamps today's rows to yesterday server-side, `POST /availability|job-availability/deactivate`, keeping free/price so the next `reactivateAll` picks them back up) before actually exiting via `getCurrentWindow().destroy()` — `destroy()`, not `close()`, since `close()` would re-emit `onCloseRequested` and loop back into this same listener. The dialog offers a third option too, "Réduire dans les icônes système" (`getCurrentWindow().hide()`, same as the existing tray minimize preference) alongside Annuler/Fermer. Only intercepts when the user is logged in — always shown past that point regardless of whether anything is actually active (skipping it silently based on hidden state made close behave unpredictably from session to session); `hasActiveAvailability` just adapts the description text and whether `deactivateAll()` is actually called.

### Notifications

Two layers, both driven by the same `onWsEvent` listener in `AuthProvider` (see "Recherche intelligente" above), not by anything route-specific — so a notification fires no matter which screen is open: an in-app `sonner` toast (always), plus — only when the main window isn't focused — a branded popup window (`src/lib/notification-window.ts`, `showNotificationPopup()`). The popup replaces `tauri-plugin-notification`: on Windows, native OS notifications only show the correct app name/icon once the app is actually *installed* (AppUserModelID registration happens via the installer's Start Menu shortcut) — in `tauri dev` they show up attributed to the launching terminal instead, which isn't fixable from app code. The popup is a small `alwaysOnTop`/`skipTaskbar`/`decorations:false` window positioned bottom-right via `primaryMonitor()`, pointed at `public/notification.html` — a **plain static page, not a router route** (it needs no auth/data/WS wiring, just the title/body passed via its URL's query string), which is why `tauri.conf.json` sets `app.withGlobalTauri: true` (exposes `window.__TAURI__` so that page can call back into the window API without going through the bundler). See `docs/notifications.md` for the full walkthrough.

## Conventions

- Path alias `@/*` → `src/*`.
- shadcn components in `src/components/ui/` are generated (`npx shadcn@latest add <name>`) — prefer regenerating/adding over hand-writing new primitives there.
- `eslint.config.js` disables `react-refresh/only-export-components` for `src/components/ui/**` and `src/routes/**` — both patterns (cva variant exports, TanStack Router's `Route` + `component` co-export) are expected there, not lint debt.
